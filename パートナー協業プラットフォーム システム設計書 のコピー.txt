1. エグゼクティブサマリー


1.1 本システムの位置づけ
本システムは、単なる「進捗管理ツール」ではない。御社がビジネスを展開する上で関わる全ての外部パートナー（代理店、協力会社、販売店、フリーランス等）との協業を、案件の種類や御社の立場に関係なく一元管理する「統合プラットフォーム」である。


1.2 解決する根本課題
* 情報の分散：案件種別ごとにExcel、スプレッドシート、メールが散在し、全体像が見えない
* 追いかけ作業の負荷：パートナーへのリマインドに担当者の時間が奪われている
* 属人化：担当者が変わると案件情報や関係者情報が引き継がれない
* 末端の不可視性：代理店の先にいる実行者の動きが見えない
* スケーラビリティ：事業やサービスが増えるたびに管理方法を作り直している


1.3 提供価値
* 時間の解放：リマインドの自動化により、追いかけ作業から担当者を解放する
* 可視化による推進力：「見られている」意識がパートナーの行動を促す
* 意思決定の高速化：ダッシュボードにより、経営・マネージャーが即座に状況を把握できる
* 組織知の蓄積：案件履歴・パートナー評価が資産として残り、属人化を防ぐ
* 無限の拡張性：新事業・新パートナーが増えても、同じ器で対応可能


1.4 投資対効果
本システムへの投資900万円に対し、以下の効果を見込む。
* 担当者1人あたり週5時間の追いかけ作業削減（年間250時間）
* 案件遅延の早期検知による損失回避（年間数百万円相当）
* パートナー評価データに基づく発注最適化
* 属人化解消による引き継ぎコスト削減

3. 解決すべき本質的課題


システム設計に入る前に、現場で実際に起きている課題を深く理解する必要がある。


3.1 情報の分散と断絶
現状の問題
* 補助金営業はExcel A、開発案件はスプレッドシートB、ASPはまた別のファイル
* パートナー情報がファイルごとに重複・不整合
* 「あの案件の情報どこだっけ？」で時間を浪費


本システムでの解決
* 全案件を1つのデータベースで管理
* 案件種別は「属性」として持ち、器は共通
* パートナー情報はマスタで一元管理


3.2 追いかけ作業の負荷
現状の問題
* 「あの件どうなりました？」のメール・電話を担当者が手動で実施
* リマインドのタイミングを覚えておく必要がある
* 担当者の時間がリマインド作業に奪われる


本システムでの解決
* リマインドを完全自動化（メール、Slack、LINE、SMS）
* エスカレーションルールを事前設定
* 担当者は「追いかける」から「確認する」だけに


3.3 階層構造の不可視性
現状の問題
* 代理店に発注したら、その先が見えなくなる
* 代理店は「順調です」と言うが、実際は止まっている
* 問題が発覚するのは炎上してから


本システムでの解決
* 関係者を階層構造（Tier 1/2/3）で可視化
* 各階層にタスクと報告義務を設定可能
* 末端からの報告も直接システムに取り込み


3.4 属人化と引き継ぎ困難
現状の問題
* 担当者の頭の中にしか情報がない
* パートナーとの関係性・過去経緯が継承されない


本システムでの解決
* 全てのアクション・コメント・報告がタイムラインに自動記録
* パートナーごとの取引履歴・評価が蓄積


3.5 スケーラビリティの欠如
現状の問題
* 新規事業を始めるたびに管理方法を考える
* パートナーが増えると管理が破綻する


本システムでの解決
* 「案件」という抽象化された器に全てを収容
* 案件種別・テンプレートを追加するだけで新事業に対応

4. 設計思想と基本方針


4.1 設計の核心思想
本システムの設計は、以下の核心思想に基づく。


「誰と、何を、いつまでに、どこまで進んでいるか」を
案件の種類や立場に関係なく、一元管理できる仕組み


4.2 3つの設計原則


原則1：抽象化による汎用性
「案件」という抽象概念を中心に据え、業務種別は案件の「属性」として扱う。これにより、どんな新規事業が始まっても同じ器で対応できる。


原則2：自動化による解放
人間が「追いかける」作業をシステムが代替する。リマインド、エスカレーション、報告催促を全て自動化。


原則3：現場第一の使いやすさ
「毎日使う平社員」と「週1で確認する上司」の両方にとって、最小の操作で最大の価値を得られるよう設計する。


4.3 現場が本当に使うための10原則


    #	原則	具体策
	1	開いた瞬間に今日やることが分かる	My Today画面をログイン直後に表示
	2	入力は最小限	必須項目3つで案件作成可能、段階的入力
	3	探さなくても見つかる	スマート検索、最近使った案件表示
	4	ワンクリックで完了できる	タスク完了、報告確認をワンクリック化
	5	通知疲れさせない	ダイジェスト配信、リマインド上限設定
	6	外部パートナーはログイン不要	メール返信、LINE、簡易フォームで報告
	7	過去の経緯が追える	タイムライン自動記録
	8	異常は自動で検知	ヘルススコア、アラート自動生成
	9	上司への報告が自動化	ワンクリックレポート生成
	10	モバイルでも使える	スマホ最適化UI
	________________


5. システム全体像


5.1 システム構成
本システムは以下の構成要素から成る。


【利用者層】
* 社内利用者：Webブラウザ、モバイルアプリからアクセス
* 外部パートナー：Webブラウザ、メール、LINE、SMSからアクセス


【アプリケーション層】
* Webアプリケーション（フロントエンド）：React + TypeScript
* APIサーバー（バックエンド）：Node.js
* リマインドエンジン：バッチ処理 + リアルタイム処理


【外部連携層】
* メール送信：SendGrid
* Slack/Teams連携：Webhook API
* LINE連携：LINE Messaging API
* SMS送信：Twilio


【データ層】
* メインDB：PostgreSQL
* ファイルストレージ：AWS S3


5.2 技術スタック
    領域	技術	選定理由
	フロントエンド	React + TypeScript	複雑なUIに対応、型安全性
	バックエンド	Node.js (NestJS)	フロントとの親和性、開発速度
	データベース	PostgreSQL	階層データの扱いに強い
	インフラ	AWS	スケーラビリティ
	認証	Auth0 or Cognito	SSO対応、セキュリティ
	メール	SendGrid	高い到達率
	

5.3 主要モジュール
* 案件管理モジュール：案件のCRUD、ステータス管理、テンプレート適用
* タスク管理モジュール：タスクのCRUD、依存関係、進捗追跡
* パートナー管理モジュール：組織・個人マスタ、階層管理、評価
* リマインドエンジン：自動リマインド、エスカレーション、ダイジェスト
* 報告収集モジュール：フォーム生成、メール取り込み、LINE連携
* ダッシュボードモジュール：可視化、アラート、レポート生成

6. データモデル設計


本システムの核となるデータモデルを定義する。


6.1 エンティティ関連図
【案件（Project）】を中心に、以下のエンティティが関連する。
* 案件 1 : N タスク
* 案件 1 : N 関係者アサイン
* 案件 1 : 1 テンプレート
* 関係者アサイン N : 1 組織
* 関係者アサイン N : 1 個人


6.2 主要エンティティ


6.2.1 案件（Project）
全ての協業の基本単位。


    属性名	型	説明
	id	UUID	一意識別子
	name	String	案件名（必須）
	project_type	Enum	案件種別（補助金/ASP/開発/自社サービス/その他）
	company_role	Enum	御社の立場（発注者/元請/営業主体/サービス提供者）
	status	Enum	ステータス（準備中/進行中/完了/中止/保留）
	health_score	Integer	自動計算されるヘルススコア（0-100）
	start_date	Date	開始日
	end_date	Date	終了予定日
	budget	Decimal	予算金額
	owner_user_id	UUID	主担当者（社内）
	custom_fields	JSON	案件種別固有の追加情報
	

6.2.2 タスク（Task）
    属性名	型	説明
	id	UUID	一意識別子
	project_id	UUID	所属案件ID
	name	String	タスク名
	status	Enum	未着手/進行中/完了/遅延/中止
	assignee_id	UUID	担当者ID
	due_date	Date	期限日
	reminder_config	JSON	リマインド設定
	

6.2.3 案件関係者（ProjectStakeholder）
    属性名	型	説明
	id	UUID	一意識別子
	project_id	UUID	案件ID
	organization_id	UUID	組織ID
	tier	Integer	階層（1=直接、2=二次、3=三次…）
	parent_stakeholder_id	UUID	上位関係者ID（階層構造用）
	

6.3 テンプレートシステム
【補助金営業テンプレート】
* フェーズ1：リード獲得（タスク：初回アプローチ、ヒアリング、案件化判断）
* フェーズ2：申請準備（タスク：書類準備、申請書作成、レビュー）

【ASP案件テンプレート】
* フェーズ1：案件受注（タスク：案件確認、見積、契約）
* フェーズ2：パートナー選定（タスク：候補選定、発注、契約）
* フェーズ3：進行管理（タスク：キックオフ、中間確認、課題対応）
* フェーズ4：納品・検収（タスク：納品確認、検収、修正対応）


【開発案件テンプレート】
* フェーズ1：要件定義（タスク：ヒアリング、要件整理、承認）
* フェーズ2：設計（タスク：基本設計、詳細設計、レビュー）
* フェーズ3：開発（タスク：実装、テスト）
* フェーズ4：リリース（タスク：受入テスト、本番リリース、検収）

7. 機能設計


7.1 案件管理機能


案件一覧
表示項目
* 案件名、案件種別（アイコン）、ステータス（色分け）
* ヘルススコア（信号機：緑/黄/赤）、主担当者
* 期限、進捗率（プログレスバー）


ビュー切替
* リスト表示：一覧性重視
* カンバン表示：ステータス別にカード表示
* カレンダー表示：期限をカレンダー上に表示


案件詳細
表示セクション
* 基本情報：案件名、種別、ステータス、期間、予算、カスタムフィールド
* 関係者ツリー：階層構造でパートナーを可視化
* フェーズ・タスク：進捗状況をタイムライン/リスト形式で表示
* タイムライン：全てのアクション・コメント・報告の履歴
* ファイル：関連ドキュメント一覧


案件作成
必須入力項目（3項目のみ）
* 案件名
* 案件種別
* 主担当者（デフォルト：自分）


7.2 タスク管理機能


タスク一覧
* 案件内タスク：特定案件のタスクを表示
* 担当者別タスク：ログインユーザーの全タスクを横断表示


タスクリマインド設定
* 予告リマインド：期限の何日前に通知するか
* 期限当日リマインド：ON/OFF
* エスカレーション：期限超過後の通知先と日数


7.3 パートナー管理機能


パートナー評価
自動計算指標
* 期限遵守率：タスク期限を守った割合
* 報告提出率：定期報告を期限内に提出した割合


手動評価（案件完了時）
* コミュニケーション：1〜5点
* 成果物品質：1〜5点
* 対応速度：1〜5点


7.4 ダッシュボード機能


My Today（個人ダッシュボード）
表示セクション
* 今日が期限のタスク：ワンクリック完了ボタン付き
* 今週が期限のタスク
* 対応待ち：自分が待っている側のタスク
* 新着報告：パートナーから届いた未確認の報告
* アラート：ヘルススコアが低下した案件


マネージャーダッシュボード
* 危険案件一覧：ヘルススコアが閾値以下
* 担当者別負荷：各担当者の案件数・タスク数
* 今週のマイルストーン

9. リマインドエンジン設計


本システムの核心機能であるリマインドエンジンの詳細設計を定義する。


9.1 設計思想
「あなたが追いかけなくても、システムが自動で追いかけてくれる」


9.2 リマインド種別
    種別	トリガー	送信先	目的
	予告リマインド	タスク期限N日前	タスク担当者	事前準備を促す
	期限リマインド	タスク期限当日	タスク担当者	当日対応を促す
	エスカレーション	タスク期限超過N日	担当者＋上位者	遅延を上位に報告
	報告催促	定期報告日	報告者	報告提出を促す
	停滞警告	案件N日間動きなし	案件担当者	放置案件を検知
	

9.3 エスカレーションルール
期限超過後、段階的に通知先を拡大していく。


    超過日数	通知先	通知強度	アクション
	1日	タスク担当者	通常	リマインドメール
	3日	担当者＋上位パートナー	強調	メール＋Slack
	7日	担当者＋上位＋案件担当	緊急	全チャネル＋社内
	14日	上記＋マネージャー	最重要	全チャネル＋アラート
	

9.4 通知チャネル
    チャネル	到達率	即時性	対象	コスト
	メール	高	中	全員	低
	Slack	高	高	社内	無料
	LINE	高	高	外部（日本）	低
	SMS	最高	最高	最終手段	高
	

9.5 リマインド最適化
通知疲れ防止策
* 同一タスクへのリマインドは最大3回まで（設定変更可）
* 「確認済み」ボタンで次のリマインドをスキップ可能


ダイジェスト配信
1日1回のダイジェストにまとめるオプション。毎朝9時配信（設定変更可）。

10. 外部パートナー向け設計


外部パートナーがストレスなく報告・確認できる仕組みを設計する。


10.1 設計原則
* ログイン不要で報告できる
* 3タップ以内で報告完了できる
* 既存の連絡手段（メール、LINE）で完結できる


10.2 報告方法


10.2.1 メール返信による報告
報告依頼メールに返信するだけでシステムに取り込まれる。


報告依頼メール例
件名：【報告依頼】案件A 週次報告（1/15締切）
案件Aの週次報告をお願いいたします。
【進捗】順調 / やや遅れ / 問題あり
【今週の実施内容】
【来週の予定】
※ 順調な場合は「順調」とだけご返信いただいても結構です。


10.2.2 簡易Webフォームによる報告
フォーム設計（3タップ完了）
* ステップ1：進捗選択（順調/やや遅れ/問題あり）→ ボタン3つ
* ステップ2：コメント入力（任意）
* ステップ3：送信


10.2.3 LINEによる報告
* システムからLINEで報告依頼メッセージを送信
* パートナーは「順調」「遅れ」等を返信
* 回答内容をシステムに自動取り込み


10.3 外部パートナー向けログイン機能
ログインが必要なケース
* 複数案件を担当しており、まとめて確認したい場合
* 下位パートナーを管理する必要がある場合


ログイン方法
* メールアドレス＋ワンタイムパスワード（マジックリンク）
* パスワード不要でセキュリティと利便性を両立

13. リスクと対策


13.1 リスク一覧
    #	リスク	影響度	発生確率	対策
	1	要件の膨張	高	高	MVPを明確化、追加要件は次フェーズへ
	2	外部パートナーの利用率低迷	高	中	ログイン不要設計、段階的導入
	3	社内利用者の定着困難	高	中	My Today設計、段階的機能開放
	4	リマインド過多	中	中	ダイジェスト機能、上限設定
	5	開発遅延	中	中	アジャイル開発、優先度明確化
	

13.2 重要リスクの詳細対策


リスク1：要件の膨張
対策
* MVP（Minimum Viable Product）を本書で明確に定義済み
* 追加要件は全て「次フェーズ検討リスト」に記録
* 予算の10%を予備費として確保


リスク2：外部パートナーの利用率低迷
対策
* ログイン不要で報告できる設計
* まずはTier 1パートナーのみで運用開始
* パートナーの声を聞き、継続的にUXを改善


リスク3：社内利用者の定着困難
対策
* My Today画面により「開いた瞬間に価値を感じる」設計
* まずは1チーム・1案件種別でスモールスタート
* 成功事例を社内に共有し、横展開

付録：用語集


用語
	説明
	案件（Project）
	本システムで管理する協業の基本単位。補助金営業、ASP、開発、自社サービス運営など全てを含む。
	タスク
	案件を構成する個別の作業単位。担当者、期限、ステータスを持つ。
	フェーズ
	案件の進行段階。テンプレートで定義され、複数のタスクを含む。
	パートナー
	御社と協業する外部の組織・個人の総称。代理店、協力会社、フリーランス等。
	Tier（階層）
	パートナーの階層。Tier 1は御社と直接取引、Tier 2はTier 1の下請け。
	リマインド
	システムから自動送信される通知。期限前の予告、期限当日、エスカレーション等。
	エスカレーション
	問題を上位者に報告すること。期限超過時に自動で上位者に通知する機能。
	ヘルススコア
	案件の健康度を0-100で自動計算したスコア。タスク遅延率等から算出。
	テンプレート
	案件種別ごとに定義された、フェーズ・タスクの雛形。
	My Today
	ログイン直後に表示される個人ダッシュボード。
	ダイジェスト
	複数の通知を1通にまとめて配信する機能。
